## Flags and things worth a second look

### 1. Migration numbering mismatch with prompt
The mission brief assumed `0006_walker_notes_realtime_publication.sql`
was already on disk. It is not. The repo's migration folders
(`/migrations/` and `/supabase/migrations/`) use date-prefixed filenames
(`20260315_...`, `20260405_...`, `20260406_...`). I followed the
explicit `0007..0011_extract_<table>.sql` naming from the brief, but
before committing you may want to either:

- rename these to `20260417_extract_<table>.sql` to match the existing
  convention, OR
- start a new `NNNN_` numbered series and backfill the earlier files.

Not my call â€” flagging.

### 2. Replica identity on four of the five tables
`walker_notes` is `REPLICA IDENTITY FULL` â€” correct, matches the fix
from the realtime gap work. The other four (`owl_notes`, `flag_cards`,
`expected_schedule`, `group_links`) are `REPLICA IDENTITY DEFAULT`.

None of those four are in the `supabase_realtime` publication today, so
**this is not a live bug.** But it's the same foot-gun that bit
walker_notes: if anyone ever adds one of these to the publication and
the client subscribes to DELETE events, the events will be silently
dropped. I did not pre-emptively flip them â€” that's a separate change.

Suggested policy for the V4 rewrite / wiggle-world port: any new table
that will ever be subscribed to via realtime should declare
`REPLICA IDENTITY FULL` in the same migration that creates it.

### 3. `expected_schedule` uses a serial sequence, not IDENTITY
Live column default is `nextval('expected_schedule_id_seq'::regclass)`.
Modern equivalent is `GENERATED BY DEFAULT AS IDENTITY`, which I used in
the migration. They are behaviourally equivalent for new inserts but
technically not byte-for-byte identical. If strict parity with the
existing sequence object matters (e.g. the sequence is referenced
elsewhere), the migration header notes the alternative spelling to use.

### 4. `expected_schedule` has no saturday/sunday columns
Only `monday..friday` exist. Possibly intentional (Wiggle doesn't walk
weekends), possibly a gap. Flagging.

### 5. `flag_cards` has both a unique constraint and a partial unique index on (source, source_id)
- `flag_cards_source_source_id_key` â€” UNIQUE constraint covering both cols
- `flag_cards_dedup` â€” UNIQUE partial index on same cols WHERE `source_id IS NOT NULL`

These overlap functionally when `source_id IS NOT NULL`. The partial
index is redundant unless the constraint was added later and nobody
dropped the index. Not a bug, just lint. Kept both in the migration to
match live state.

### 6. Two styles of RLS on `owl_notes`
The table has both anon policies (`Allow anon read/insert owl_notes`)
and a full set of authenticated policies (`owl_notes_select/insert/
update/delete`). Anon can read and insert freely. If the HQ port
tightens the security model, the anon policies are the first things to
revisit.

### 7. `flag_cards` policy role = `public`
`"Admins can manage flag_cards"` applies to `public` role rather than
`authenticated`. Same pattern as `"Admins can update walker_notes"`.
Effectively equivalent because the `qual` requires `auth.uid()` to
match a profile row, but `TO authenticated` would be tidier and signal
intent more clearly. Kept `TO public` to match live state.

### 8. FK `ON DELETE / ON UPDATE` are all `NO ACTION`
All four FKs (`walker_notes.dog_id â†’ dogs.id`,
`owl_notes.target_dog_id â†’ dogs.id`, `flag_cards.dog_id â†’ dogs.id`,
`flag_cards.actioned_by â†’ profiles.id`) use `NO ACTION` for both
delete and update. This means deleting a parent row whose children
still reference it will error. For dogs that churn (e.g. dog
deprecated, re-added), this may surface as a delete failure in Tower.
Not changing anything â€” flagging.
